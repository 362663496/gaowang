# 设计接口访问权限管理

## Goal

为系统补齐可信登录态与基于固定角色的细粒度接口访问控制，使管理员能安全配置 `staff` 权限，并让后端与前端始终按同一权限结果工作。

## Background

- 用户希望能够为某些角色设置接口访问权限。
- 当前阶段只做需求澄清和技术规划，不实施业务代码。

## Confirmed Facts

- 现有角色固定为 `admin` 与 `staff`，用户创建接口也只接受这两个值（`apps/api/internal/models/models.go:11`、`apps/api/internal/http/handlers/users.go:51`）。
- 当前后端把接口分成“任意已登录用户”和“仅 admin”两组；用户、备份、设置、审计接口属于 admin 组（`apps/api/internal/http/router.go:24`、`apps/api/internal/http/router.go:62`）。
- 当前认证中间件直接信任客户端传入的 `X-Dev-User-ID` 和 `X-Dev-Role`，未校验签名，也未从数据库加载用户与角色（`apps/api/internal/http/middleware.go:14`）。
- 前端把用户 ID 和角色保存在 `localStorage`，并随请求发送上述头部；调用者可以自行伪造 `admin`（`apps/web/src/lib/api.ts:5`、`apps/web/src/lib/api.ts:88`）。
- `AUTH_SECRET` 已作为必填配置加载，但尚未参与认证（`apps/api/internal/config/config.go:18`）。原始系统设计要求安全 Cookie 会话或签名令牌。
- 前端菜单权限与后端路由分组并不完全一致：staff 仍能看到备份和设置入口；API 客户端又把 `403` 当作登录失效并清空会话。

## Requirements

- 本任务包含替换当前可伪造的开发头部登录态；接口授权必须基于服务端可验证的用户身份和服务端读取的当前角色。
- 登录使用服务端数据库会话，不使用 JWT；浏览器只持有随机会话令牌，数据库仅保存令牌哈希、用户 ID、过期时间和必要审计时间。
- 会话令牌仅通过 `HttpOnly`、生产环境 `Secure`、`SameSite` Cookie 传递，不再把用户 ID、角色或认证令牌保存在 `localStorage`。
- 会话 Cookie 仅用于同源网页请求；修改类请求必须校验同源 `Origin`，不额外引入 CSRF 令牌或前端安全状态。
- 会话有效期全系统固定为 7 天，使用持久 Cookie 跨浏览器重启保持，不提供用户自选期限或“记住我”选项。
- 首版认证仅服务于仓库内置 Next.js 网页客户端，不提供第三方程序、移动端或脚本使用的 Bearer Token/JWT。
- 同一账号允许多个并发会话；普通退出仅撤销当前会话，修改密码或账号被数据库标记停用时撤销或立即拒绝该用户全部会话。
- 新安装且 `users` 表为空时，必须通过一次性环境变量 `INITIAL_ADMIN_NAME`、`INITIAL_ADMIN_EMAIL`、`INITIAL_ADMIN_PASSWORD` 创建首个 `admin`；已有用户后引导逻辑不得再次创建或覆盖账号，也不开放匿名注册接口。
- 权限变更必须记录操作者、变更前权限、变更后权限和时间到现有审计日志，不记录会话令牌或密码。
- 权限必须在后端接口层强制执行，不能只依赖前端隐藏入口。
- 授权判断集中在统一 Gin 中间件中；业务 handler 不得自行查询角色或编写权限分支。
- 每个受保护路由必须显式关联一个业务权限键，路由声明是策略映射，实际放行或拒绝由中间件统一执行。
- 首版复用项目现有 Gin 与 GORM 实现轻量授权，不引入 Casbin、OPA 或其他通用策略引擎。
- 权限使用稳定的业务动作键表达，由后端路由映射到权限；不把 HTTP 方法与路径作为管理员直接配置的权限标识。
- 删除等不可逆业务动作必须具备独立权限，不能隐含在普通新增或编辑权限中。
- 首版权限按独立业务操作细分；商品明确拆为查看、新增、编辑、启停、删除五项，写操作分别授权并自动依赖查看权限。
- 首版权限目录如下：
  - 商品：`product.read`、`product.create`、`product.update`、`product.toggle`、`product.delete`。
  - 店铺：`shop.read`、`shop.create`。
  - 库存：`inventory.read`、`inventory.inbound`、`inventory.sales_outbound`、`inventory.adjust`。
  - 库存流水：`movement.read`。
  - 报表：`report.sales_summary`、`report.sales_trend`、`report.product_ranking`、`report.shop_ranking`。
  - 审计日志：`audit.read`。
  - 备份：`backup.read`、`backup.run`。
  - 系统设置：`setting.read`、`setting.update`。
  - 用户：`user.read`、`user.create`，固定为 `admin` 专属。
  - 权限：`permission.read`、`permission.update`，固定为 `admin` 专属。
- 仪表盘不设置独立权限，只展示当前用户有权读取的数据块。
- `/uploads/*` 商品图片继续公开访问，不受 `product.read` 限制；商品、库存和价格等结构化数据仍由 API 权限保护。
- 首版沿用固定的 `admin` 与 `staff` 两种角色，不支持创建、改名或删除角色。
- `admin` 永远拥有全部权限且不可被削权；仅 `staff` 的权限可配置。
- 所有 `staff` 用户共享同一套角色权限，不支持用户级允许或拒绝覆盖。
- 用户管理与权限管理永久为 `admin` 专属，不得授予 `staff`。
- 用户管理保持现有“查看、新建”范围，允许 `admin` 新建其他 `admin`；本次不新增停用、改角色或删除用户的接口与界面。
- 服务端每次认证都校验账号启用状态；若账号被数据库标记为停用，其已有会话不得继续使用。
- 审计日志、备份和系统设置作为互相独立的权限，可由管理员授予 `staff`；查看与执行或修改分开授权。
- 管理员通过网页权限矩阵配置 `staff` 权限，配置持久化到数据库并在运行时生效。
- 权限矩阵按模块与操作分组，列出 `admin` 与 `staff` 两列；`admin` 权限全部勾选且锁定，`staff` 权限可编辑，依赖权限自动勾选并明确标注，整套 `staff` 权限一次性保存。
- 权限系统首次启用时，`staff` 默认不拥有任何业务权限；必须由管理员显式勾选后才能访问。
- 权限存在依赖关系；勾选操作权限时自动补齐必要的读取权限，移除基础权限时同步移除依赖它的权限。
- 后端是权限依赖关系的唯一数据源，保存时必须拒绝未知或 `admin` 专属权限并计算依赖闭包；前端根据同一目录实现自动勾选与级联移除。
- 首版权限依赖为：商品写操作依赖 `product.read`；`shop.create` 依赖 `shop.read`；`inventory.read` 依赖 `product.read`；入库与销售出库依赖 `inventory.read` 和 `shop.read`；库存调整依赖 `inventory.read`；`movement.read` 依赖 `product.read` 与 `shop.read`；`backup.run`、`setting.update`、`user.create`、`permission.update` 分别依赖对应的读取权限。
- 新增到代码目录但未由管理员选择的权限默认拒绝 `staff`，升级不得自动扩权。
- 零权限 `staff` 仍可登录，并可查看账号、修改自己的密码和退出；界面显示无可用功能的明确空状态。
- `401` 表示登录失效并清除会话；`403` 仅表示当前功能无权访问，不得清除会话。
- 权限修改从下一次后端请求起立即生效；前端不使用实时推送，而是在窗口重新获得焦点、页面刷新或收到 `403` 后重新读取当前权限。
- 无页面查看权限时隐藏对应菜单；有页面查看权限但无操作权限时隐藏对应操作按钮；直接访问无权页面时显示 403 页面且保留登录态。

## Acceptance Criteria

- [ ] 未获授权的角色直接请求受保护接口时会被拒绝。
- [ ] 已获授权的角色可以继续正常调用对应接口。
- [ ] 仅拥有商品新增、编辑、启停权限但没有 `product.delete` 的 `staff` 调用删除接口时得到 `403`，其他已授权动作不受影响。
- [ ] 客户端伪造用户 ID 或角色不能提升权限。
- [ ] 数据库中不保存可直接重放的明文会话令牌，浏览器脚本无法读取会话 Cookie。
- [ ] 已删除或已失效的会话不能继续调用受保护接口。
- [ ] 非同源修改请求在到达业务 handler 前被拒绝。
- [ ] 新数据库可以安全创建唯一的首个管理员，重复启动不会重复创建。
- [ ] 每次权限更新都能在审计日志中追溯操作者及前后差异。
- [ ] 权限配置不能导致系统失去拥有完整权限的管理员入口。
- [ ] `staff` 无法通过用户管理或权限管理接口创建管理员、修改授权或自行提权。
- [ ] 同一时刻所有已启用 `staff` 对同一权限键得到一致的授权结果。
- [ ] 管理员保存 `staff` 权限后，无需重新部署即可影响后续接口请求。
- [ ] 新安装或从旧版本升级后，未配置的 `staff` 请求全部受保护业务接口时均被拒绝。
- [ ] 系统不能保存会导致已授权页面无法加载的残缺权限组合。
- [ ] 自动化检查能发现任何未关联权限策略的新受保护业务路由。
- [ ] 零权限 `staff` 登录后不会进入重定向循环，也不会因业务接口返回 `403` 被强制退出。
- [ ] 已登录用户重新聚焦页面或遇到拒绝响应后，菜单和操作项会与最新权限一致。
- [ ] 最终方案明确权限粒度、角色来源、配置方式和默认策略。

## Out of Scope

- 自定义角色及其新增、改名、删除和用户迁移流程。
- 商品图片的鉴权下载或签名 URL。
- 第三方 API Token、OAuth、JWT 或移动端认证。
- 用户停用、角色变更、用户删除等完整账号生命周期管理。
