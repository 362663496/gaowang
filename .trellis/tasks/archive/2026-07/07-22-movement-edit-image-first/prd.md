# 流水编辑与商品图片优先交互

## Goal

允许有权限的操作员安全修正某个商品的最新库存流水，使数量和价格修正真实更新库存与财务结果，同时把商品主图提升为选择、识别和确认商品的首要信息。

## Background

- 操作员主要通过图片识别商品，对名称和编码不敏感；现有文字下拉和部分纯文字列表容易选错商品。
- 录入错误目前只能追加新的库存操作，流水页没有编辑能力；用户需要在保留责任链的前提下修正最新一笔记录。
- 本需求已通过逐项需求访谈确认，首版采用严格边界，避免任意历史重放和多图系统。

## Confirmed Facts

- 流水接口当前只有分页查询，页面明确描述为“不可变库存流水”（`apps/api/internal/http/handlers/movements.go:15`、`apps/web/src/app/(app)/stock-movements/page.tsx:82`）。
- 入库、销售出库和库存调整都通过 `InventoryService` 在事务中更新 `InventorySnapshot` 并追加 `StockMovement`；流水保存了数量、采购/销售价格、成本、收入和毛利的派生结果（`apps/api/internal/services/inventory.go`）。
- `StockMovement` 当前没有修订版本、最后编辑人或最后编辑时间（`apps/api/internal/models/models.go:108`）。
- 现有权限只有 `movement.read`，并依赖商品和店铺读取权限（`apps/api/internal/services/permissions.go:24`、`apps/api/internal/services/permissions.go:61`）。
- 商品选择器当前是名称/编码文字下拉，主图组件已提供缺图占位和加载失败回退（`apps/web/src/features/product-combobox.tsx:7`、`apps/web/src/features/product-image.tsx:8`）。
- 新建商品当前允许省略图片，图片服务已限制为 JPG/JPEG/PNG/WebP 且不超过 5MB（`apps/api/internal/http/handlers/products.go:64`、`apps/api/internal/services/uploads.go`）。
- 商品销售排行响应没有图片字段，仪表盘和部分报表仍以纯文字显示商品（`apps/api/internal/http/handlers/reports.go:79`、`apps/web/src/app/(app)/dashboard/page.tsx`、`apps/web/src/app/(app)/reports/page.tsx`）。

## Requirements

### 流水编辑范围

- 仅允许编辑该商品按原始时间和流水 ID 排序后的最新一条流水；后端是“是否最新”的唯一判断方。
- 非最新流水显示禁用编辑入口，并解释“仅可编辑该商品最新一条流水”；分页或筛选不得由前端自行推断最新记录。
- 商品、流水类型、原操作人和原始时间保持锁定；不增加流水删除能力，也不支持把一条流水改成另一种类型或另一个商品。
- 入库可修改：正数数量、非负进货单价、可选店铺、可选备注。
- 销售出库可修改：正数出库数量、非负销售单价、必填店铺、可选备注。
- 库存调整可修改：非零有符号调整数量、必填备注；调整流水没有店铺。
- 备注最长 500 字。每次编辑另外必填最长 500 字的“修改原因”，修改原因只进入审计，不覆盖业务备注。
- 只改备注或店铺时，只更新元数据和审计，不制造库存数量或金额变化。
- 已归档商品的最新流水只允许修改备注和店铺；数字字段保持锁定。
- 不增加删除入口；错误流水统一通过受控编辑修正。

### 库存与财务语义

- 编辑保留原流水 ID、原操作人和原始时间；原行更新，完整原值由审计日志保存。
- 入库数量变化影响库存数量、库存金额和移动平均成本；入库单价变化影响库存金额和移动平均成本。
- 销售出库数量变化影响库存数量、收入、成本和毛利；销售单价变化只影响收入和毛利。
- 库存调整数量变化影响库存数量和按当时成本计算的库存金额。
- 店铺或备注变化不影响库存和财务数字。
- 成本单价、采购金额、收入、成本金额和毛利均为服务端派生字段，客户端不得直接提交最终值。
- 编辑销售出库后，修正后的收入、成本和毛利继续计入原流水日期所属的报表周期。
- 保存前展示服务端计算的影响预览：原值与新值、当前与结果库存、库存金额变化，以及适用时的收入、成本和毛利变化。

### 一致性、审计与权限

- 新增独立、可授予 staff 的 `movement.update` 权限，并依赖 `movement.read`；拥有新增库存操作权限不自动获得改账权限。
- 保存时在单个数据库事务内再次确认目标仍是最新流水、版本仍匹配、库存结果合法，然后更新库存快照、流水、最后编辑信息和审计日志。
- 编辑窗口打开后若出现更新的流水或同一流水已被他人修改，返回 `409 MOVEMENT_STALE`，要求刷新后重试。
- 扩大销售出库后库存不足、调整后库存为负、输入数字越界或其他不合法结果时，整笔编辑返回稳定错误，原流水、库存和审计均不变化。
- 每次成功编辑记录完整 `before`、`after`、影响结果、修改原因、编辑人和编辑时间；页面显示“已修改”、最后编辑人和最后编辑时间。

### 商品图片优先

- 新建商品必须上传一张主图；服务端和前端都校验。历史无图商品继续可查询和操作，但显示醒目的“待补图”占位。
- 保持单主图模型，沿用 JPG/JPEG/PNG/WebP、最大 5MB；不增加多图、相册或在线裁剪。
- 所有商品选择入口统一为图片宫格选择器：80–96px 主图优先，名称和编码作为辅助信息与搜索文本。
- 入库、销售出库、库存调整、流水筛选和流水编辑复用同一个商品选择组件；当前选中项也必须展示图片。
- 商品列表、库存、流水、仪表盘最近流水、低库存、商品排行及商品相关确认框统一使用共享商品身份组件；普通表格/列表图片尺寸为 40–48px，不再出现纯文字商品项。
- 商品排行 API 增加当前 `product_image_path`；历史流水、报表和其他只读视图始终展示商品当前主图，不保存交易时图片快照。
- 图片缺失或加载失败时使用一致的方形占位，并保留可读的名称、编码和“待补图”文字，不能只靠图片传达信息。

## Acceptance Criteria

- [ ] 有 `movement.update` 的用户可打开最新流水编辑；无该权限的用户看不到可用编辑操作。
- [ ] 非最新流水由服务端标记为不可编辑，直接请求保存时得到冲突响应。
- [ ] 修改最新入库的数量或单价后，库存数量、金额和移动平均成本与从修正值重新执行该操作的结果一致。
- [ ] 修改最新销售出库的数量或售价后，库存、收入、成本、毛利和原日期报表同步正确。
- [ ] 修改最新调整数量后，库存不为负且金额按该操作时的成本正确更新。
- [ ] 仅修改备注或店铺不会改变库存或财务数字。
- [ ] 商品、流水类型、原操作人和原始时间无法通过请求修改。
- [ ] 并发新流水或过期版本使保存得到 `409 MOVEMENT_STALE`，且没有部分写入。
- [ ] 库存不足、负库存或非法数值使整笔编辑失败，原状态不变。
- [ ] 已归档商品的数字编辑被拒绝，但其最新流水可修正备注或店铺。
- [ ] 每次成功编辑产生一条包含完整前后值、影响、修改原因、编辑人和时间的审计记录。
- [ ] 编辑过的流水显示修订状态、最后编辑人和时间；原操作人和时间仍可见。
- [ ] 数字保存前展示服务端计算的影响预览，保存时仍重新校验最新状态和版本。
- [ ] 新建商品缺少图片时，前端阻止提交，直接调用 API 也得到稳定校验错误。
- [ ] 历史无图商品仍可操作，并在选择器和列表中显示“待补图”。
- [ ] 所有商品选择器以图片宫格展示并支持名称/编码搜索、清空和键盘聚焦。
- [ ] 商品列表、库存、流水、仪表盘、低库存、商品排行和确认框均以图片为首要商品标识。
- [ ] 商品换图后，历史流水和报表显示新主图；排行响应包含 `product_image_path`。
- [ ] 自动化测试覆盖权限、最新判断、三类重算、并发冲突、事务回滚、审计、图片必填、图片选择搜索与缺图回退。

## Out of Scope

- 编辑任意历史流水或重放目标之后的多条流水。
- 删除流水、修改商品、修改流水类型、修改原操作人或原始时间。
- 自定义角色、用户级权限覆盖或按流水类型拆分多个编辑权限。
- 多图相册、历史图片快照、在线裁剪、图片识别或相似图搜索。
- 按店铺拆分实际库存；店铺仍是流水归属与报表维度。
- 恢复已归档商品或允许归档商品重新产生库存。
